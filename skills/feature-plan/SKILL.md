# Feature Plan — 任务规划师

将 `design.md` 拆解为可逐条执行的任务清单（`tasks.md`），为后续 `rc:feature-execute` 或 `rc:plan-executor` 提供明确输入。

## 设计说做什么，任务说怎么做

`design.md` 定义"系统应该变成什么样"，`tasks.md` 定义"按什么顺序、改哪些文件、写什么代码"来实现它。

## 工作流程

### Step 1: 阅读设计

1. **读取 CLAUDE.md**：了解项目架构边界、代码规范、验证命令
2. **读取 design.md**：如果用户提供了路径，直接读取；否则搜索 `docs/features/{module}/{version}/design.md` 或 `docs/features/{module}/{version}/{side}/design.md`
3. **读取 analysis.md**（如果存在）：补充需求背景
4. 确认设计文档完整（front-matter、核心章节），如果不完整，提醒用户先完成设计

### Step 2: 拆解任务

按以下原则拆解：

#### 拆解维度：按文件

每个任务对应一个或一组紧密相关的文件变更。不要把多个不相关文件的改动合并到一个任务中。

#### 依赖顺序

严格按依赖关系排序：

```
数据模型 / Schema → 服务层 / 业务逻辑 → 路由 / UI → 配置 / 收尾 → 验证
```

#### 任务粒度规则

| 规则 | 说明 |
|------|------|
| 单文件优先 | 一个任务最好只涉及一个文件 |
| 子任务 ≤ 5 | 超过 5 个子步骤就拆成多个任务 |
| 代码骨架 | 每个任务包含关键函数/类的签名和注释，不写完整实现 |
| 明确输入输出 | 每个任务标注依赖哪个前置任务的产出 |

#### 代码骨架深度

- **新建文件**：给出文件结构、类/函数签名、关键注释
- **修改文件**：标注修改位置、修改前后的差异要点
- **配置文件**：给出需要添加/修改的配置项

### Step 3: 双轨注册

1. **写入 tasks.md**：输出到 `docs/features/{module}/{version}/tasks.md`；如果 design.md 位于 `{side}/` 子目录下，则 tasks.md 也写入同一子目录（`docs/features/{module}/{version}/{side}/tasks.md`）
2. **TODO 同步**（可选）：如果用户要求，提醒用户手动将任务同步到项目 TODO 系统

### Step 4: 与用户确认

向用户呈现：
1. 任务总数和预估工作量
2. 依赖关系图（简要文字描述或 Mermaid）
3. 红线列表（从 design.md 的"暂不实现"提取）
4. 是否有需要用户决策的拆解歧义

等待用户确认后定稿。

## 状态标记

| 标记 | 含义 |
|------|------|
| ⬜ | 待处理 |
| 🔧 | 执行中 |
| ✅ | 已完成 |
| ⚠️ | 验证未通过 |

## 边界规则（HARD STOP）

**本 Skill 的职责到 tasks.md 定稿为止。** 用户确认任务清单后，必须立即停止，不得继续执行以下任何操作：

- ❌ 不写代码、不创建源文件、不修改源代码
- ❌ 不安装依赖、不运行构建命令
- ❌ 不启动子代理去实现任何代码变更
- ❌ 不自行开始执行任务清单中的任何一条任务

### 完成后提示用户

任务清单确认完成后，输出以下提示（替换 `{module}` 为实际模块名）：

```
✅ tasks.md 已定稿，共 {n} 个任务。

🚀 下一步：运行 /rc:feature-execute {module} 开始逐任务执行（每个任务完成后会有验证门控）。
   或运行 /rc:plan-executor docs/features/{module}/v1/tasks.md 进行批量自动执行。
```

## 原则

1. **design 说做什么，tasks 说怎么做**：不要在 tasks.md 中重复设计决策，聚焦于"怎么改代码"
2. **依赖顺序是铁律**：数据模型永远先于服务层，服务层先于路由/UI
3. **代码骨架不是完整实现**：给出足够的结构让执行器理解意图，但不替代实际编码
4. **红线清单必须从 design.md 提取**：不要自行增减"暂不实现"的范围
5. **任务粒度适中**：太粗导致执行器无法自主完成，太细导致管理开销过大
6. **用中文输出**：所有文档内容使用中文
