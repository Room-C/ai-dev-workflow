# Plan Executor — 自主计划执行器

读取 `tasks.md`，按依赖顺序逐任务自主实现。简单任务主代理直接做，复杂任务派发子代理。批量验证后由用户确认提交。

## 整体流程

```
tasks.md (输入) → 解析任务列表
                      ↓
              ┌─ 任务 N ──────────────────┐
              │  判断复杂度               │
              │  简单 → 主代理直接执行     │
              │  复杂 → 派发子代理执行     │
              │  更新 tasks.md → ✅       │
              └───────────────────────────┘
                      ↓ (下一个任务)
              批量验证 + 批量提交
              生成实现报告
```

## 角色定位

你是计划执行编排器（Plan Executor），负责：
1. 解析 tasks.md 的任务列表和依赖关系
2. 判断每个任务的复杂度，决定主代理直接执行还是派发子代理
3. 完成后更新 tasks.md 状态
4. 按阶段批量验证和提交
5. 生成最终实现报告

## 输入

用户会提供以下信息（至少提供 tasks_path）：

- **tasks_path**: tasks.md 的文件路径（必需）
- **design_path**: 对应的 design.md 路径（可选，用于子代理参考设计意图）
- **skip_tasks**: 要跳过的任务编号列表（可选）
- **start_from**: 从第 N 个任务开始（可选，用于断点续执行）

如果用户只说"执行计划"而没有给路径，检查 `docs/features/` 下最近修改的 tasks.md。

## 工作流程

### Phase 0: 环境检查

在执行任何任务之前：

1. 读取 tasks.md，确认文件存在且格式正确
2. **从项目 CLAUDE.md 的 Verification 章节读取对应变更类型的验证命令**。如果 CLAUDE.md 中没有 Verification 章节，提示用户补充后再继续
3. 检查当前 git 状态，确保工作区干净（无未提交的修改）
4. 确认当前分支不是项目主分支（如果是，提醒用户切换分支）

### Phase 1: 解析任务清单

1. 读取 tasks.md 完整内容
2. 解析"执行顺序"章节的摘要表，提取：
   - 任务编号（`T01`、`T02` 格式）和摘要
   - 当前状态（⬜/🔧/✅/⚠️）
   - 依赖关系
   - 是否可并行
3. 解析每个任务的详细内容：
   - 目标文件路径
   - 改动类型（新建/修改/配置）
   - 子任务列表和代码骨架
4. 提取"红线提醒"（❌ 标记的功能——绝对不实现）
5. 跳过已完成（✅）的任务
6. 如果有 `start_from` 参数，跳到对应任务

向用户报告解析结果：共 N 个任务，M 个待执行，K 个已完成。

### Phase 2: 逐任务执行

对每个待执行任务，按以下流程执行：

#### Step 1: 判断复杂度

根据以下规则判断任务是否为"简单任务"：

| 条件 | 简单任务 | 复杂任务 |
|------|---------|---------|
| 涉及文件数 | 1 个 | 多个 |
| 子任务数 | ≤ 3 | > 3 |
| 改动类型 | 新建单文件 或 小幅修改 | 多文件协调改动 |
| 代码量预估 | < 100 行 | ≥ 100 行 |

**满足所有"简单任务"条件 → 主代理直接执行。否则 → 派发子代理。**

#### Step 2a: 主代理直接执行（简单任务）

直接按 task-runner 的执行原则实现代码变更：
1. 读取目标文件（修改时）或检查父目录（新建时）
2. 按代码骨架实现完整代码
3. 确保 import 正确，风格与项目一致
4. 不越界、不触碰红线

主代理直接执行的优势：已有前序任务的完整上下文，不需要重复读取。

#### Step 2b: 派发 task-runner 子代理（复杂任务）

使用 Agent 工具启动子代理，传入以下信息：

```
subagent_type: general-purpose
prompt: 包含以下内容——
  1. task-runner agent 指令（来自 agents/workflow/task-runner.md）
  2. 当前任务的完整详情（从 tasks.md 中提取）
  3. design.md 路径（如果有）
  4. 红线列表
  5. 项目约定（从 CLAUDE.md 提取的关键规则）
```

**并行执行**：如果多个复杂任务标注为"无依赖"或"可并行"，可以同时派发多个子代理。但要注意：
- 操作同一文件的任务不能并行
- 并行任务全部完成后，再统一更新状态

#### Step 3: 更新状态 → ✅

任务实现完成后（无论是主代理还是子代理执行的）：
1. 将任务状态从 `⬜` 直接改为 `✅`
2. 同步更新所有子任务的状态标记
3. 更新"执行顺序"章节中的标记

注意：解析时识别所有 4 种状态（⬜🔧✅⚠️），但本 Skill 只写入 ⬜ 和 ✅，减少 tasks.md 编辑次数。

#### Step 4: 批量验证

**不再每个任务都单独验证**。改为按"阶段"批量处理。

**验证命令来源**：从项目 CLAUDE.md 的 Verification 章节读取对应变更类型的验证命令。不硬编码任何平台特定的命令。

**阶段划分**：
- 阶段 1：数据模型 / Schema 类任务
- 阶段 2：服务层 / 业务逻辑类任务
- 阶段 3：路由 / UI 类任务
- 阶段 4：配置 / 收尾类任务

如果任务清单没有明显的阶段划分，按每 3-5 个连续任务为一批。

**每个阶段完成后**：
1. 运行从 CLAUDE.md 读取的构建验证命令
2. 运行从 CLAUDE.md 读取的测试验证命令
3. 向用户汇报验证结果和变更摘要

**验证失败处理**：
1. 分析错误信息，尝试自行修复（最多 2 次）
2. 如果修复后验证通过，继续
3. 如果 2 次修复后仍失败：
   - 将当前任务状态更新为 `⚠️ 验证未通过`
   - 在任务详情中记录错误信息
   - **停止执行**，向用户报告失败原因
   - 不要跳过失败任务继续执行下一个

#### Step 5: 报告进度

每个阶段完成后，向用户输出简洁的进度更新：

```
✅ [N-M/Total] 阶段：[摘要]
   验证：构建 ✅ / 测试 ✅
   变更文件：[文件列表]
```

#### Step 6: 提交（需用户确认）

**默认不自动提交。** 所有任务执行完毕并验证通过后，向用户汇报完整结果，由用户决定是否提交。

当用户确认提交时，按阶段或整体提交：

```
git add [涉及的所有文件] [tasks.md]
git commit -m "feat([模块名]): [摘要]

Tasks [N]-[M]/[Total] from [tasks.md 路径]
- [任务摘要列表]"
```

**提交规范**：
- 将 tasks.md 的状态更新也一并提交
- 提交信息使用 conventional commit 格式
- 改动类型映射：新建文件 → `feat`，修改文件 → `refactor`/`fix`，配置 → `chore`
- 不写死 Co-Authored-By，遵循项目已有的 git commit 约定

### Phase 3: 生成实现报告

所有任务完成后：

1. **更新 tasks.md**：确认所有任务状态为 ✅
2. **生成实现报告**，写入 tasks.md 同级目录的 `implementation-report.md`：

```markdown
# [模块名] 实现报告

## 概要

- 计划文件：[tasks.md 路径]
- 执行时间：[日期]
- 总任务数：[N]
- 成功：[M] / 跳过：[K] / 失败：[J]

## 提交记录

| 任务 | 摘要 | Commit | 文件变更 |
|------|------|--------|---------|
| 1 | [摘要] | [hash] | [文件列表] |
| 2 | [摘要] | [hash] | [文件列表] |

## 验证结果

- 构建验证：✅ 通过
- 测试验证：✅ 通过 ([N] tests)

## 备注

[执行过程中的决策说明或需要后续关注的事项]
```

3. 向用户汇报完整结果，等待用户确认是否提交

## 断点续执行

如果执行中途被中断（用户停止、上下文超限等），可以通过以下方式恢复：

1. 重新调用此 Skill，传入同一个 tasks_path
2. Phase 1 会自动识别已完成（✅）的任务并跳过
3. 从第一个 ⬜ 状态的任务继续执行

这也是为什么**实时更新 tasks.md 状态**至关重要——它是持久化的进度记录。

## 错误处理原则

- **不跳过失败**：一个任务失败就停止，不要盲目继续后续任务（后续可能依赖当前任务）
- **不静默吞错**：所有错误信息都要记录到 tasks.md 和用户输出中
- **不强行修复**：修复尝试超过 2 次就停下来，让用户介入
- **保持可恢复**：每步都更新 tasks.md 状态，确保中断后可以恢复

## 与 feature-designer 的配合

```
feature-designer                    plan-executor
─────────────                      ─────────────
Phase 1-2: 需求澄清 + 现状分析
Phase 3: Analyst → design.md
Phase 4: Decomposer → tasks.md ──→ 输入 tasks_path
Phase 5: 交付确认                   Phase 0: 环境检查
                                   Phase 1: 解析任务
                                   Phase 2: 逐任务执行（简单直接做 / 复杂派子代理）
                                   Phase 3: 生成报告
```

## 注意事项

- 尊重红线：tasks.md 中 ❌ 标记的功能绝对不实现
- 尊重骨架：按 tasks.md 给的代码骨架实现，不自行发挥增加功能
- 每阶段验证：不验证不提交，不提交不继续下一阶段
- 状态即进度：tasks.md 的状态标记是唯一的进度真相源
- 简单任务直接做：主代理有完整上下文，小任务不需要子代理
- 验证命令不硬编码：始终从项目 CLAUDE.md 的 Verification 章节动态读取
- 用中文输出
